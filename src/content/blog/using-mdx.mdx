One of the advantages of using Astro for this blog is MDX support. MDX is a format that lets you write JSX directly inside Markdown files, which means you can embed interactive components alongside your prose.

## Why MDX?

Standard Markdown is great for text-heavy content, but sometimes you need more. Maybe you want to include an interactive demo, a styled callout, or a reusable component that renders differently based on props.

MDX bridges that gap. You write Markdown as usual, and when you need a component, you import and use it inline:

```mdx
import MyChart from './MyChart.astro';

Here's some regular Markdown text.

<MyChart data={salesData} />

And the text continues below the chart.
```

## When to Use MDX vs Plain Markdown

For most blog posts, plain `.md` files are the better choice. They're simpler, faster to write, and have no JavaScript overhead. I use MDX only when a post genuinely needs embedded components — interactive examples, styled alerts, or data visualizations.

A good rule of thumb: if the post reads fine as plain text, use Markdown. If it needs interactivity or custom rendering, reach for MDX.

## Setting It Up in Astro

Astro's MDX integration is straightforward. Add the plugin to your config:

```javascript
import mdx from '@astrojs/mdx';

export default defineConfig({
  integrations: [mdx()],
});
```

Then create `.mdx` files in your content directory. Astro handles the compilation automatically.

## A Note on Performance

By default, Astro renders MDX components as static HTML with zero client-side JavaScript. If you need a component to be interactive in the browser, you add a client directive:

```mdx
<InteractiveWidget client:load />
```

This opt-in model keeps your pages fast. Only the components that genuinely need JavaScript will ship it.

MDX is a powerful tool, but like any tool, it's best used deliberately. Start with plain Markdown and upgrade to MDX when you have a clear reason to.
